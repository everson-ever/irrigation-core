## Title

Migrate persistence layer from JSON Lines files to a normalized SQLite database

## Specification

Replace the current JSON Lines file storage (`data/schedules.json`, `data/valves.json`,
`data/settings.json`, `data/history.json`) with a single SQLite database
(`data/irrigation.db`). The database schema must normalize the data currently
flattened into JSON records — most importantly `Schedule.weekdays`, which is
stored today as a JSON array embedded in a single JSON-Lines record and must
become its own table. All reads and writes (list, get by id, insert, update,
delete, bulk replace) must use parameterized SQL statements executed through a
new repository implementation that satisfies the existing `Repository`
protocol (`src/irrigation/domain/ports.py`), so `application/services.py` and
the `domain` layer stay unchanged.

`data/history_search_results.json` is a derived, transient artifact (a
materialized snapshot of the last history search, written only so Node-RED's
"historico pesquisado" `file in` node can display it) and is not authoritative
data. It stays a JSON file, regenerated by `HistoryService.search_range` after
every search, same as today.

### Context

The system is a Raspberry Pi irrigation controller made of a long-running
Python daemon (`irrigation run`), one-off `irrigation` CLI invocations
triggered by Node-RED `exec` nodes (create/update/delete schedules, turn
valves on/off, update settings, search history), and a Node-RED dashboard that
also reads three files directly with `file in` nodes: `data/valves.json`,
`data/settings.json`, and `data/history_search_results.json`.

`JsonLinesRepository` (`src/irrigation/infrastructure/json_repository.py`)
currently guarantees safety across these concurrent readers/writers using
`fcntl` advisory locks, atomic `os.replace`, a stat-based read cache, and
torn-write repair for crash safety mid-append. It works, but every "table" is
a flat file where each line is a full JSON object, ids are computed by
scanning all records for `max(int(id)) + 1`, `weekdays` is a JSON array
embedded inside a schedule record instead of being normalized, and range
queries (`HistoryService.search_range`) load the entire history file into
Python and filter it there, which gets slower as `history.json` grows without
bound over the years. Moving to SQLite gives ACID transactions and real
locking (WAL + `busy_timeout`) instead of hand-rolled file locking, lets the
schema express relationships (schedule → weekdays) properly, and lets range
queries use indexes instead of full scans, while keeping insert/update/delete
semantics identical from the application layer's point of view.

### Scope

#### In scope

- Design a normalized SQLite schema for schedules, schedule weekdays, valves,
  settings, and history.
- Implement SQLite-backed repositories that satisfy the `Repository` protocol
  used by `application/services.py` (`list_all`, `find_by_id`, `add`,
  `update`, `delete`, `replace_all`).
- Preserve the current external contract of each repository: `id` is returned
  as a string, `add` assigns the next id, `update` raises
  `RecordNotFoundError` for unknown ids, `delete` is idempotent and returns
  whether anything changed, `replace_all` overwrites the full table
  transactionally.
- One-time migration of existing `data/*.json` content into
  `data/irrigation.db` for installations that already have data (Raspberry Pi
  deployments), run automatically the first time the app starts and no
  database file exists yet.
- Add CLI read commands for valves and settings (`irrigation valve list`,
  `irrigation settings show`) so Node-RED's `file in` nodes for
  `data/valves.json` and `data/settings.json` can be replaced by `exec` nodes,
  matching how every write already flows through the CLI.
- Update `node-red/flows.json` to read valves and settings through the new
  `exec`-based commands instead of `file in` nodes pointed at JSON files that
  will no longer exist.
- Update `deploy/data-defaults/`, `scripts/install-raspberry.sh`, `README.md`,
  and `deploy/package-readme.md` to describe the SQLite-based persistence and
  the new default database instead of default JSON files.
- Rewrite `tests/test_repository.py` for the SQLite repository behavior, and
  update `tests/test_cli.py`/`tests/test_services.py` fixtures and assertions
  that currently read/write `data/*.json` files directly.

#### Out of scope

- Changing `data/history_search_results.json`'s role as a transient,
  Node-RED-facing snapshot file — it is not moved into SQLite.
- Any change to domain rules (schedule validation, duplicate-valve rejection,
  manual/automatic irrigation logic) beyond what is required to keep behavior
  identical on top of the new storage.
- Adding a full migrations/versioning framework (e.g. Alembic). A single
  `schema.sql`/`CREATE TABLE IF NOT EXISTS` bootstrap plus the one-time JSON
  import is enough for this project's size.
- Changing the Node-RED flow for history writes/search (`history` `exec`
  node already returns data on stdout; only the two direct `file in` reads for
  valves and settings are replaced).

## Technical approach

### Design principles

- Keep each class and function focused on one responsibility.
- Depend on abstractions at architectural boundaries.
- Keep domain rules independent from infrastructure details.
- Prefer small, explicit interfaces and simple data flows.
- Avoid speculative abstractions, duplicated logic, and unrelated changes.

### Proposed changes

1. **Schema** (`src/irrigation/infrastructure/sqlite_repository.py`, executed
   as `CREATE TABLE IF NOT EXISTS` on first connection):

   ```sql
   CREATE TABLE IF NOT EXISTS schedules (
       id               INTEGER PRIMARY KEY AUTOINCREMENT,
       time             TEXT    NOT NULL,           -- "HH:MM"
       duration_minutes INTEGER NOT NULL CHECK (duration_minutes > 0),
       valve_pin        INTEGER NOT NULL CHECK (valve_pin > 0),
       status           INTEGER NOT NULL DEFAULT 0 CHECK (status IN (0, 1)),
       enabled          INTEGER NOT NULL DEFAULT 1 CHECK (enabled IN (0, 1))
   );

   CREATE TABLE IF NOT EXISTS schedule_weekdays (
       schedule_id INTEGER NOT NULL REFERENCES schedules(id) ON DELETE CASCADE,
       weekday     TEXT    NOT NULL CHECK (weekday IN
                       ('mon','tue','wed','thu','fri','sat','sun')),
       PRIMARY KEY (schedule_id, weekday)
   );

   CREATE TABLE IF NOT EXISTS valves (
       id                  INTEGER PRIMARY KEY AUTOINCREMENT,
       pin                 INTEGER NOT NULL UNIQUE,
       section             TEXT    NOT NULL,
       status              INTEGER NOT NULL DEFAULT 0 CHECK (status IN (0, 1)),
       manually_turned_off INTEGER NOT NULL DEFAULT 0 CHECK (manually_turned_off IN (0, 1))
   );

   CREATE TABLE IF NOT EXISTS settings (
       id                       INTEGER PRIMARY KEY CHECK (id = 1),
       default_duration_minutes INTEGER NOT NULL CHECK (default_duration_minutes > 0)
   );

   CREATE TABLE IF NOT EXISTS history (
       id      INTEGER PRIMARY KEY AUTOINCREMENT,
       valve   TEXT NOT NULL,   -- section name at run time (kept denormalized:
                                 -- history must not change if a valve is later
                                 -- renamed or removed)
       date    TEXT NOT NULL,   -- ISO date
       start   TEXT NOT NULL,   -- "HH:MM"
       end     TEXT NOT NULL,   -- "HH:MM"
       weekday TEXT NOT NULL,
       mode    TEXT NOT NULL
   );
   CREATE INDEX IF NOT EXISTS idx_history_date ON history(date);
   ```

   `history.valve` stays a denormalized text snapshot (not a `valves.id`
   foreign key) on purpose: history records must remain accurate even if the
   valve's `section` name changes or the valve row is deleted later, and
   `HistoryService` is never given a valve id, only a section string.
   `settings` stays a single-row table (`CHECK (id = 1)`), matching
   `SettingsService`'s existing "first row or insert" semantics.

2. Implement `SqliteRepository(connection, table)` for `valves`, `settings`,
   and `history` — generic `list_all`/`find_by_id`/`add`/`update`/`delete`/
   `replace_all` built from the table's column list, each read returning
   `id` as `str(...)` to match `Schedule.from_dict`/`Valve.from_dict` calling
   `str(data.get("id"))`.

3. Implement `ScheduleSqliteRepository(connection)` separately: `add`/`update`
   write the `schedules` row plus the matching `schedule_weekdays` rows inside
   one transaction (delete-then-insert the weekday rows on update, since the
   set is small and this avoids diffing); `list_all`/`find_by_id` join
   `schedule_weekdays` back into a `weekdays` list ordered by `WEEKDAY_IDS`
   before returning the dict, so `Schedule.from_dict` sees the same shape it
   sees today.

4. Every write path (`add`, `update`, `delete`, `replace_all`) runs inside a
   single `sqlite3` transaction (`BEGIN IMMEDIATE` / commit / rollback on
   error) so a crash mid-write cannot leave a schedule without its weekdays,
   replacing what `fcntl` + atomic replace did for the JSON files. Open the
   connection with `PRAGMA journal_mode=WAL`, `PRAGMA foreign_keys=ON`, and a
   `busy_timeout` (e.g. 5000ms) so the daemon and short-lived CLI invocations
   from Node-RED don't fail on transient contention.

5. `src/irrigation/infrastructure/json_migration.py`: on `Application.create`
   (`bootstrap.py`), if `data/irrigation.db` does not exist yet and any of the
   legacy `data/*.json` files does, open/create the database, run the schema
   bootstrap, then import each JSON Lines file into its table (schedules
   import also fans each record's `weekdays` array out into
   `schedule_weekdays`), preserving the original `id` values so existing
   Node-RED-side references keep working. This runs once; after migration the
   JSON files are left untouched on disk (not deleted) so the operator can
   confirm the import before removing them manually.

6. `src/irrigation/config.py`: replace `Settings.file_path(name)` with
   `Settings.database_path` (`data_dir / "irrigation.db"`).
   `src/irrigation/bootstrap.py`: open one shared `sqlite3.Connection` in
   `Application`, and have `schedules()`, `valves()`, `runtime_settings()`,
   `history()` build the corresponding repository from it instead of a
   `JsonLinesRepository` per file. `history()` still builds a second,
   JSON-file-backed repository for `history_search_results.json` (unchanged).

7. `src/irrigation/cli.py`: add `valve list` (returns
   `app.valves().list_all()` as dicts) and `settings show` (returns the
   current settings row) subcommands/handlers, following the existing
   `_COMMAND_HANDLERS`/`create_parser` pattern.

8. `node-red/flows.json`: replace the `file in` node reading
   `data/valves.json` (id `8d0b3804.e60cf8`) and the one reading
   `data/settings.json` (id `d19f016a.a6ac8`) with `exec` nodes running
   `irrigation valve list` / `irrigation settings show`, wired to the same
   downstream nodes those `file in` nodes currently feed
   (`a51e1954.c69208`, `a7ddf74d.7238a8`). Leave the
   `data/history_search_results.json` `file in` node as is.

9. Update `README.md` (persistence section, "File names and JSON fields are
   part of the current dashboard contract", the per-file bullet list around
   line 290) and `deploy/package-readme.md` (default-data section) to
   describe the SQLite schema and the single remaining JSON contract file
   (`history_search_results.json`).

10. `scripts/install-raspberry.sh` and `deploy/data-defaults/`: ship a seed
    step (or a pre-built empty `irrigation.db` with the default
    `settings` row and no valves, matching today's "ship with an empty
    `valves.json`" behavior) instead of default `valves.json`/
    `settings.json`/`schedules.json`/`history.json`.

### Performance considerations

- Expected data size: a handful of valves and schedules, history growing by a
  few rows per irrigation event — realistically low thousands of rows over
  the device's lifetime. SQLite comfortably handles this.
- `HistoryService.search_range` today loads the whole `history.json` into
  Python and filters with a list comprehension: `O(n)` per search regardless
  of the requested range. With `history(date)` indexed, the equivalent
  becomes `SELECT * FROM history WHERE date BETWEEN ? AND ?`, `O(log n + k)`
  where `k` is the result size.
- `add()` no longer needs to scan every existing id to compute
  `max(id) + 1` (`O(n)` in the current JSON implementation); `INTEGER
  PRIMARY KEY AUTOINCREMENT` makes this `O(1)`.
- Avoid opening a new SQLite connection per repository call from the
  long-running daemon (`IrrigationController.run`); reuse one connection per
  `Application` instance, matching the current one-`JsonLinesRepository`
  -per-process lifetime. Each short-lived CLI invocation still opens and
  closes its own connection, same as today's process-per-`exec`-node model.

### Error handling and edge cases

- `update()` on an unknown id must keep raising `RecordNotFoundError` (checked
  today by `test_update_missing_id_fails`) — check row existence before/within
  the transaction and roll back if absent.
- `delete()` must stay idempotent, returning `False` and leaving the table
  unchanged when nothing matched the given ids (`ON DELETE CASCADE` on
  `schedule_weekdays` must not raise when a schedule has zero weekday rows,
  which cannot happen given the domain's `_normalize_weekdays` guarantee but
  should not be assumed at the SQL layer).
- Concurrent writers (daemon + Node-RED-triggered CLI invocations) must not
  see "database is locked" errors under normal load — covered by WAL mode and
  `busy_timeout`; add a regression test that runs two repository instances
  against the same file and interleaves writes (mirroring
  `test_cache_sees_writes_from_other_process`).
- `SettingsService.default_duration_minutes()` must keep raising
  `ValidationError` when no row exists yet — the `settings` table starts
  empty (no default row is auto-inserted), same as `settings.json` starting
  empty until `update_default_duration` or `deploy/data-defaults` seeds it.
- Migration from legacy JSON: malformed/partial legacy JSON (e.g. the torn
  final line case the current `JsonLinesRepository` tolerates) must not
  silently drop data or crash startup — reuse
  `JsonLinesRepository`'s existing tolerant read for the import step instead
  of re-parsing the files by hand.
- Migration must be run at most once: presence of `data/irrigation.db` is the
  sole signal to skip re-importing, so a second startup after migration never
  re-inserts rows or duplicates ids.

## Notes

- `history_search_results.json` is intentionally kept as a JSON file — it is
  a query result snapshot for Node-RED's display, not source-of-truth data,
  and folding it into SQLite would add a table with no writer other than "the
  last search," which brings no normalization benefit.
- The legacy `data/*.json` files are left on disk after migration rather than
  deleted, so an operator can diff/verify before removing them by hand; a
  follow-up task can automate cleanup once this migration has been verified
  on real Raspberry Pi installs.
